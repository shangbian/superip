# 新旧工作流结构与请求参数对比

## 结论概览

| 项目         | 旧工作流（v1）     | 新工作流（v3/chat）   |
|--------------|--------------------|------------------------|
| **是否可用** | ✅ 可通（配置正确时） | ❌ 当前报错：Missing required parameters |
| **触发条件** | 配置了 `COZE_WORKFLOW_ID`，未配 `COZE_BOT_ID` | 配置了 `COZE_BOT_ID` |
| **API 地址** | `POST https://api.coze.cn/v1/workflows/chat` | `POST https://api.coze.cn/v3/chat` |
| **响应方式** | 一次返回完整结果（或 SSE 流） | 先返回任务 id，再轮询 retrieve + message/list 取结果 |

---

## 一、结构差异

### 1. 旧工作流（v1/workflows/chat）

- **模型**：一个「工作流」对应一个 `workflow_id`，工作流内部通过 `parameters.choice` 等做多智能体路由。
- **调用链**：前端 → 本后端 → Coze **v1/workflows/chat** → 一次拿到 output + conversation_id。
- **会话**：通过 `conversation_id` 续聊，历史放在 `additional_messages`。

### 2. 新工作流（v3/chat）

- **模型**：一个「Bot」对应一个 `bot_id`，没有 choice 概念，每个 Bot 独立。
- **调用链**：前端 → 本后端 → Coze **v3/chat**（创建任务）→ 轮询 **v3/chat/retrieve** → 完成后调 **v3/chat/message/list** 取回复。
- **会话**：同样用 `conversation_id`，历史在 `additional_messages`。

---

## 二、请求参数差异（核心）

### 旧工作流请求体（v1）

```json
{
  "workflow_id": "7599232922009583626",
  "conversation_id": "可选，续聊时传",
  "parameters": {
    "USER_INPUT": "用户输入内容",
    "choice": 1,
    "topic": "主题，一般同 USER_INPUT"
  },
  "additional_messages": [
    { "content": "...", "content_type": "text", "role": "user", "type": "question" },
    { "content": "...", "content_type": "text", "role": "assistant", "type": "answer" },
    { "content": "当前用户输入", "content_type": "text", "role": "user", "type": "question" }
  ]
}
```

- **必填**：`workflow_id`、`parameters`（含 `USER_INPUT`、`choice`、`topic`）、`additional_messages`。
- **parameters** 是工作流内部定义的入参，用于路由和逻辑。

### 新工作流请求体（v3）

```json
{
  "bot_id": "7616269899295916066",
  "user_id": "user_xxx 或 固定值",
  "stream": false,
  "conversation_id": "可选，续聊时传",
  "additional_messages": [
    { "content": "...", "content_type": "text", "role": "user", "type": "question" },
    { "content": "当前用户输入", "content_type": "text", "role": "user", "type": "question" }
  ],
  "parameters": {}
}
```

- **必填**：`bot_id`、`user_id`、`additional_messages`。
- **parameters** 当前在代码里传的是 **空对象 `{}`**。若该 Bot 在扣子平台配置了必填变量，就必须在这里传入，否则会报 **Missing required parameters**。

---

## 三、差异点小结

| 维度         | 旧工作流（v1）                          | 新工作流（v3）                                |
|--------------|-----------------------------------------|-----------------------------------------------|
| 身份标识     | `workflow_id`                           | `bot_id`                                      |
| 用户标识     | 无单独字段                              | `user_id` 必填                                |
| 业务入参     | `parameters.USER_INPUT / choice / topic` | `parameters` 当前为空，需按 Bot 配置补全      |
| 多智能体     | 靠 `parameters.choice` 在工作流内路由   | 无 choice，一个请求对应一个 Bot               |
| 历史消息     | `additional_messages`                   | `additional_messages`（结构一致）             |
| 会话延续     | `conversation_id`                       | `conversation_id`                              |
| 返回方式     | 一次返回或 SSE                          | 先返回任务，再轮询 retrieve + message/list    |

---

## 四、当前状态与原因

- **旧工作流**：在只配 `COZE_WORKFLOW_ID` + 旧 Token 时，可以通。请求里带了工作流需要的 `USER_INPUT`、`choice`、`topic`，所以不会报缺参。
- **新工作流**：配了 `COZE_BOT_ID` + 新 Token 后，请求里 **没有** 给该 Bot 在扣子平台配置的必填参数（都在 `parameters` 里，而我们传的是 `{}`），所以 Coze 返回 **Missing required parameters**，属于 Bot 侧必填参数未传，不是接口地址或 Token 问题。

---

## 五、新工作流要打通可以怎么做

1. **在扣子平台看该 Bot 的必填参数**  
   打开 Bot `7616269899295916066`，看「对话」或「工作流」里定义的必填输入变量（名字、类型）。

2. **在本后端把参数传进 `parameters`**  
   在 `executeV3Chat` 里构造 `parameters` 时，把上述必填变量都带上。例如若 Bot 要求 `user_input`、`topic`，可以类似：

   ```ts
   parameters: {
     user_input: userInput.trim(),
     topic: (topic && topic.trim()) ? topic.trim() : userInput.trim()
   }
   ```

   具体字段名要以扣子平台里该 Bot 的配置为准。

3. **若 Bot 不要求额外参数**  
   在扣子平台把该 Bot 的必填项改成非必或删掉，这样当前 `parameters: {}` 也可以不报错。
